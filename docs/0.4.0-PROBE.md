# 0.4.0 Probe Report — Spellcasting recon

Date: 2026-06-16 · By: Claude (Opus 4.8), live capture in the GM client (user "Dade", who is
`game.users.activeGM`) via in-page evaluation on **Foundry 13.350 / PF1 11.11**. Resolves
`RECON_FINDINGS.md` open probes **#2** (`pf1PreActionUse` payload/timing) and **#5**
(player-facing spell-chat suppression feasibility), plus the 0.4.0 probe list in
`ROADMAP.md`/`PLAN.md`. **This is a probe/report milestone only — NO masking or feature code was
written, and none should be until 0.5.0.** Mirrors the proven `0.3.0-PROBE.md` pattern.

## Method

NPC spellcasting is GM-controlled, so the recon is single-client (the Vivaldi/GM seat); no
cross-client socket was needed for the probe itself. On the live GM client I:

1. Armed temporary capture hooks (`pf1PreActionUse`, `pf1PreDisplayActionUse`,
   `preCreateChatMessage`, `createChatMessage`, `pf1PostActionUse`) that stashed snapshots to a
   `globalThis.__pf15probe` var with a shared monotonic **sequence counter** (to prove ordering)
   and recorded `game.messages.size` at each step (to prove the card did not yet exist).
2. Cast **Magic Missile** from the NPC **Oswald** (`hasPlayerOwner: false`) via
   `item.use({ skipDialog: true })`.
3. Read the captured payloads and inspected the resulting `ChatMessage` document (the replicated
   surface a player can read).
4. **Restored the world:** removed all temp hooks + the capture global, restored Oswald's Magic
   Missile prepared slot (`preparation.value` 1→0→1), and deleted my own test card. `messages.size`
   returned to its pre-probe baseline (85). No other user's message was touched.

Source cross-check: PF1 `ActionUse` in `V:\FoundryVTTData\Data\systems\pf1\pf1.js` (bundled),
which confirms the runtime ordering below at the code level.

### Where spell identity lives on the single card (replicated surface)

The one `type:"action"` ChatMessage bundles **identity and already-public mechanics together**:

- **Identity (leaks):** rendered `content` (~18 KB) contains the spell name in plain text (card
  title **"Magic Missile (Use)"**), plus `data-item-id`, `data-action-id`, `data-token-uuid`
  attributes; `message.system.item.id` holds the spell item id; at `preCreateChatMessage` the
  `flags.pf1.metadata.item` carried `{ id, description, identified:true }`.
- **Mechanics (already public by design):** the same card embeds **6 action buttons**
  (attack/damage/save/apply controls) and `message.system` keys `rolls, targets, save, action`.
- There is **no separate mechanics-only message** to preserve. Saves/damage become **separate
  follow-up messages only when a user later clicks the card's buttons** (post-cast,
  user-initiated). A spell that uses a measure template places a **canvas `MeasuredTemplate`
  document** (not a chat message); Magic Missile uses none.

## Suppression feasibility (probe #3 / RECON #5) — characterization only

**Verdict: feasible and leak-free for non-player (GM-controlled) casts, via GM-side
pre-creation interception — but the single-card bundling is the real design problem, not the
hook.**

Interception points, ranked:

| Point | Effect | Verdict for masking |
|-------|--------|---------------------|
| **A. `pf1PreActionUse` → `false`** | Cancels the ENTIRE action: no card, **no mechanics**, deletes the measure template. | **Too blunt.** Also kills the saves/damage the table is entitled to. Useful only as the earliest *detection* hook, not the suppression mechanism. |
| **B. `pf1PreDisplayActionUse` → `false`** (or set `shared.scriptData.hideChat = true`) | Mechanics/charges/script-calls have already run; `ChatMessage.create` is skipped. **The real card document is never created** → nothing to leak, nothing to delete. PF1-native, GM-side. | **Best PF1-native option.** Lets the system finish its bookkeeping, then withholds the card. |
| **C. Foundry `preCreateChatMessage` → `false`** | Prevents the document being written on the **creating client** (the GM, for NPC casts). System-agnostic. | **Good universal fallback.** Prevents creation; deletes nothing. |
| **D. `renderChatMessageHTML` (per-client DOM)** | Blanks/replaces the card's DOM for non-owners. | **Reject for privacy.** The document still holds the real name/description/ids on the replicated surface — a player reading `game.messages.get(id).content` / `.system.item.id` in console sees it. Presentation-only, **not** confidential. |

**Why this satisfies both hard constraints.** NPC casts run on the GM client (`pf1PreActionUse`
fired on the active-GM seat), and the card is created by the GM and *then* broadcast. So the
active GM intercepts **its own** creation (B or C) **before** the document reaches the database or
any other client. Therefore: (1) **no other user's message is ever deleted** — there is no
other-user message; the GM cancels its own pre-creation; and (2) **the original is never leaked**
— the real card is never created on a replicated surface.

**The genuine hard part (flag for 0.5.0 design).** Because identity and mechanics share one card,
suppressing the card also withholds the public attack/save/damage. Options:

- *Generic-only (simplest, matches Decision #5 = name+school first):* post "A spell is being cast"
  with **no** mechanics until a Spellcraft success. Trade-off: the otherwise-public attack roll /
  save prompt is withheld until identification.
- *Stripped-mechanics re-post (harder):* the GM re-posts a card that surfaces save DC / damage /
  attack result with the **name/school/description removed**. More work and more leak surface to
  audit (button labels, flags, `system.item.id`, data-attributes must all be scrubbed).

**Scope boundary.** This clean GM-side pre-emption only holds for **non-player casters**
(`hasPlayerOwner === false` → GM-controlled → cast originates on the GM client). If a **player**
casts (their own PC), `pf1PreActionUse` fires on the *player* client and the player is the
creator/owner; the GM cannot pre-empt that creation without the player's client self-suppressing
(a socket round-trip). 0.5.0 scope (PLAN Decision #4) is non-player casters only, so this is out
of scope — but it is the natural future limit. A measure-template spell also drops a canvas
template that the card-suppression does not remove (it can telegraph the spell's area/type).

## Spellcraft key / rank (probe #4)

- **Key:** `pf1.config.skills.spl` = `"Spellcraft"` (confirmed at runtime, not merely from the
  lang file). Matches how 0.3.0 used `per`.
- **Rank path:** `actor.system.skills.spl.rank` (sub-keys `ability, rt, acp, rank, mod, cs`),
  exactly parallel to `per`.
- **Eligibility differs from Perception:** `spl.rt === true` (trained-only). 0.3.0 found
  Perception usable untrained, so eligibility there is "player-owned actor". For Spellcraft the
  identifier must have **`spl.rank >= 1`** in addition to ownership. (Confirm the table's exact
  "eligible identifier" rule with Michael before the 0.5.0 request UI, as in 0.3.0.)

## Non-player-caster detection (probe #5 / PLAN Decision #4)

- **Discriminator:** `actionUse.actor.hasPlayerOwner === false`, read directly on the hook payload.
- **`actor.type` does NOT work here:** the live spellcasting NPCs (Oswald, Reginald) are
  `type:"character"` yet `hasPlayerOwner:false`; the player PC "Pants" is also `type:"character"`
  but `hasPlayerOwner:true`. A `type === "npc"` check would miss these casters. `hasPlayerOwner`
  is the correct, PLAN-aligned test.

## Implications for the 0.5.0 Spell Identification MVP

The 0.5.0 build reuses the proven 0.3.0 socketlib request/broker + active-GM private store. No
new transport or storage primitive is required — only a spell-side gate and the suppression step.

1. **Detect (active-GM client):**
   ```js
   Hooks.on("pf1PreActionUse", (au) => {
     if (au?.item?.type === "spell" && au?.actor && !au.actor.hasPlayerOwner && isActiveGMClient()) { … }
   });
   ```
   Stash the real identity at this point (it is the earliest, pre-card hook).
2. **Suppress (GM-side, leak-free):** use **B** (`pf1PreDisplayActionUse` / `scriptData.hideChat`)
   or **C** (`preCreateChatMessage`) to stop the real card on the GM's own client. Do **not** use
   A (it kills mechanics) and do **not** rely on D (render-time DOM — it leaks). Stash the true
   `{spellName, spellItemUuid, casterActorUuid, originalChatData}` in the **active-GM private
   store** (already client-scope, world-partitioned, non-replicated; these keys are already in
   `FORBIDDEN_PUBLIC_KEYS`). Post a generic public card via the public registry:
   `spellcasting[castId] = { state:"masked", identifiedBy:{}, publicLabel:"A spell is being cast." }`
   — `assertNoSecrets` already guards that surface.
3. **Mint a cast id:** the `actionUse` has **no native per-cast id** (only item/action ids, which
   repeat across casts). The module must synthesize a stable `castId`, e.g.
   `` `${actorId}:${itemId}:${seq-or-worldTime}` ``.
4. **Identify (reuse 0.3.0 broker):** GM requests Spellcraft from eligible PCs (eligibility =
   **player-owned actor with `spl.rank >= 1`**, because `spl.rt === true`); player rolls;
   `pf1ActorRollSkill` captures `chatMessage.rolls[0].total` with `skillId === "spl"` (0.3.0's
   confirmed capture, swapping `"per"`→`"spl"`); relay total to the active GM via socketlib;
   broker compares to a hidden Spellcraft DC in the private store; on success reveal **name +
   school** (Decision #5) — personal first, GM globalizes (Decision #6). DC never leaves the GM.
5. **Decide the mechanics question up front (the central design call):** because saves/damage are
   bundled into the one suppressed card, the MVP should post the **generic notice only** (withhold
   mechanics until identification) for v1, with a "stripped-mechanics re-post" deferred. Confirm
   with Michael.
6. **Honor the boundaries:** non-player casters only; player self-casts and measure-template
   telegraphing are explicit out-of-scope limits to document, not silently mask.
7. **Privacy gate (every release):** player-seat reload + inspect the masked-cast `ChatMessage`
   (`content`, `system.item.id`, `flags`, data-attributes), the public registry, and socket
   payloads — confirm no spell name/description/uuid reaches any replicated surface. The single
   card's plain-text title and `system.item.id` are the specific leak surfaces to verify are
   absent.
