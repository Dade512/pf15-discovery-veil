# 0.6.0 Runtime Verification — Shared Discovery UI (GM panel)

Status: **PASSED — live, active-GM client (2026-06-16).** Driven via in-page evaluation on the live
GM client (Vivaldi, world `baphomets-fall-dev`, scene "Staging"), Foundry `13.350` / PF1 `11.11`.
All touched state was restored afterward (gates + my private-store entries cleared, the one reveal
card deleted, both test tokens returned to visible, panel closed). Built on the 0.5.0 state layer and
the confirmed-working 13.350 ApplicationV2 / `getSceneControlButtons` pattern (baphomet-utils
`weather-ui.js`).

## What was verified

  `main.mjs:21` (new line), `pf15DiscoveryVeil.version === "0.6.0"`, status

  `pf15-discovery-veil.openPanel` is registered.
- The token **scene-control button** registers: `ui.controls.controls.tokens.tools.pf15dvPanel`

### Render (PASS)
Seeded two perception gates (Oswald DC 18 spotted by Pants/DXWarlock; Boots DC 22, no one) and two
masked spells (Fireball lvl 3 → DC 18, no one; Charm Person lvl 1 → DC 16, identified by Pants).
The panel rendered:
- **Undetected creatures (2)** — each row: token name, `scene · state · DC`, a "Spotted by:" line
  (names or "no one yet"), and the action buttons.
- **Masked spell casts (2)** — each row: the **true spell name** (active-GM only), `Masked · school
  · DC` (schools localized: Evocation / Enchantment; DCs 15 + level), an "Identified by:" line, and
  the action buttons.
- A **Refresh** footer button. State-dependent buttons confirmed (an undetected gate shows
  Manage/Request/Reveal/Focus/Clear; after global reveal the same gate shows only Focus/Clear).

### Privacy (the point of the module — PASS)
- The seeded **public registry contains no secrets**: a full `JSON.stringify` of the registry has no
  "Fireball", no "Charm", no `"dc"`, no school code — only safe state (`state`, `spottedBy` /
  `identifiedBy` with safe attribution, `publicLabel`). `assertNoSecrets` holds.
- **No secret in any `data-*` attribute**: every `[data-action]` button carries only
  `action` + the public ids (`sceneId` / `tokenId` / `castId`, all random ids). No name / school / DC.
- The true spell names, schools, and DCs are read only through the **active-GM-gated getters**
  (`getHiddenSpellIdentity` / `getHiddenPerceptionDC`), which fail closed to `null` off the active GM
  (proven cross-client in 0.5.0). The panel is GM-only by three independent guards: `openDiscoveryPanel`
  returns for non-GMs, `_renderHTML` bails for non-GMs, and the scene-control/keybinding are GM-only.
  No new replicated surface is introduced — the panel is a read/control view over already-verified state.

### Live refresh (PASS)
With the panel open, a registry mutation (`markSpotted` adding Jaden to the Boots gate) caused the
panel to **re-render automatically** — Boots' status line updated to "Spotted by: Jaden's PC (Jaden)"
with no manual refresh. This is the `updateSetting` hook path; a real player's successful Spellcraft/
Perception roll updates the same registry via the active-GM broker (proven in 0.3.0/0.5.0) and fires
the identical hook, so a player success appears on the GM's open panel the same way.

### Actions (PASS — real ApplicationV2 dispatch via `button.click()`)
- **perReveal** (Oswald): gate → `globallyRevealed`; native token hidden restored to its prior value
  (token became visible). Delegates to the proven `revealGloballyAction`.
- **spellReveal** (Fireball): gate → `globallyRevealed`; posted the public card "The spell is
  identified: Fireball (Evocation)." (`whisper:[]`). Delegates to `revealSpellGlobally`.
- **perClear** (Boots): public gate removed; native token hidden restored (visible). Delegates to
  `clearGateAction`.
- **spellClear** (Charm Person): public gate removed **and** the private identity cleared
  (`getHiddenSpellIdentity` → `null`). Uses the new `clearSpellMask` (public + active-GM private).
- The dialog-opening actions (Manage spotted / Request check / Request Spellcraft) reuse the
  0.2.0/0.3.0/0.5.0 DialogV2 flows unchanged; their buttons were confirmed present and correctly
  wired (`data-action`), not re-opened in automation.

## Notes / by-design / follow-ups
- **Verified on the active-GM seat only.** The panel is a GM-only surface with no player-side
  component; the one cross-client aspect (live refresh) was exercised via a registry mutation, which
  is the same `updateSetting` path a remote client's change triggers. Player-seat privacy is covered
  by the three GM-only guards plus the 0.5.0 cross-client fail-closed proof of the same getters, so a
  separate two-client pass was judged unnecessary for this milestone.
- **Housekeeping quirk — now fixed in 0.6.0.** Previously, clearing a perception gate
  (`clearGateAction`, shared with the 0.2.0 HUD) removed the public gate but left the token's hidden
  DC in the active-GM private store (localStorage). `clearGateAction` now also calls
  `setHiddenPerceptionDC(..., null)` (and the panel's orphaned-token clear branch does the same), so a
  cleared gate leaves no orphaned secret behind — mirroring how `clearSpellMask` clears both surfaces.
  On a non-active GM the private write is a harmless no-op (the DC only ever lived on the active GM).
  Re-checked live after the fix: clearing a freshly-gated token leaves the private bucket clean. The
  one pre-existing orphaned entry observed during the first pass (an unrelated Goblin token from an
  earlier session) was left untouched, as it is the user's data, not a test artifact.
