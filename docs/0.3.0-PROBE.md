# 0.3.0 Probe Report — Perception roll capture

Date: 2026-06-15 · By: Claude (Opus 4.8), live capture in the GM client (user "Dade") via
in-page evaluation on **Foundry 13.350 / PF1 11.11**. Resolves `RECON_FINDINGS.md` open probes
**#1** (`pf1ActorRollSkill` payload/timing) and **#3** (rank path / eligibility). Probes #2/#5
(spell recon) remain for the 0.4.0 milestone.

## Method

Armed `Hooks.on("pf1ActorRollSkill", (...args) => stash primitives)` in the live client, then
invoked `actor.rollSkill("per", { skipDialog: true })` on a player-owned actor ("Pants") and read
the captured payload. (The first manual console run was lost to console-tracking timing, so this
was re-captured into a variable.)

## Confirmed findings

| Item | Result |
|------|--------|
| Hook signature | `pf1ActorRollSkill(actor, chatMessage, skillId)` — exactly 3 args |
| Arg types (live) | `ActorCharacterPF`, `ChatMessagePF`, `String` |
| **Roll total location** | **`chatMessage.rolls[0].total`** (test roll returned `22`) |
| `skillId` form | plain key string, e.g. `"per"` (3rd arg) |
| Timing | Hook fires **after** the ChatMessage is created and its `rolls` are populated — i.e. the message is passed in complete. Good for read-only capture. |
| Firing client | Fires on the client that **calls `rollSkill`** (the roller). A player roll fires it on the player client → 0.3.0 must capture player-side and relay the total to the active GM via socket (matches the PLAN §C broker). |
| Roll API | `actor.rollSkill(skillId, options)` confirmed; `{ skipDialog: true }` is honored (no dialog). |
| Skill keys | `pf1.config.skills.per` → "Perception", `…spl` → "Spellcraft". |
| Rank path | `actor.system.skills.per.rank` (sub-keys: `ability, rt, acp, rank, mod, cs`). |

## Eligibility — refinement of the PLAN

PLAN.md §D/0.3.0 sketched eligibility as `actor.system.skills.per.rank >= 1` + ownership. The
live actor "Pants" has **`per.rank = 0`** yet rolls Perception fine (total 22) — Perception is a
**non-trained-only** skill, usable untrained in PF1. So a `rank >= 1` gate would wrongly exclude
valid Perception rollers.

**Recommendation:** for Perception, eligibility = **player-owned actor** (`actor.hasPlayerOwner`),
not a rank threshold. Keep an optional rank/“can-roll” refinement for trained-only skills later
(relevant to Spellcraft in 0.5.0, which IS trained-only). To be confirmed by Michael before the
request UI is built.

## Implications for the 0.3.0 build

- Capture: `Hooks.on("pf1ActorRollSkill", (actor, chatMessage, skillId) => …)` on each client;
  when `skillId === "per"` and the user has an open request for a gated token, read
  `chatMessage.rolls[0].total` and emit it to the active GM over `module.pf15-discovery-veil`.
- Broker (active GM only): compare the relayed total to `getHiddenPerceptionDC(sceneId, tokenId)`
  (now active-GM gated); on `total >= dc` call `markSpotted(... source:"perception")`. DC never
  leaves the GM. No automatic global reveal.
- The GM needs a way to set the hidden DC when marking undetected (wire `setHiddenPerceptionDC`).
- No spell/Spellcraft work here (gated to 0.4.0).
