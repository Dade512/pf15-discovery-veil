# 0.1.1 Spike Report — pf15-discovery-veil

Date: 2026-06-15 · Runtime spikes by Claude (Opus 4.8), driving Michael's live Vivaldi (GM)
and Chrome (player) seats remotely (Michael at work, authorized in chat). Foundry 13.350 /
PF1 11.11. The world was restored to its pre-spike state afterward (one test chat message may
remain in the log).

## Probe 1 — `pf1ActorRollSkill` payload & timing → RESOLVED

- Runtime-captured signature: **`pf1ActorRollSkill(actor, chatMessage, skillId)`** — 3 args:
  `ActorPF`, `ChatMessagePF`, and the skill id `String` ("per"). This matches the
  `CLAUDE_HANDOFF.md` assumed signature.
- The `ChatMessagePF` carries `rolls` and `systemRolls`, so the roll total is readable at
  **`chatMessage.rolls[0].total`** (confirm the exact index/path when building 0.3.0).
- Timing: the hook fires together with a constructed `ChatMessagePF` (the result message already
  exists when the hook runs).
- The roll was triggered with `actor.rollSkill("per", { skipDialog: true })`; `skipDialog:true`
  suppressed the roll dialog cleanly. One Perception chat message was posted as a test artifact.
- Implication for 0.3.0: capture totals by listening to `pf1ActorRollSkill` and reading
  `chatMessage.rolls[0].total`; the active-GM broker compares it to the hidden DC.

## Probe 4 — per-client token visibility → RESOLVED (confirms Decision #2)

Tested on the goblin token (restored afterward).

- **Native `hidden` base:** with `token.hidden = true`, the PLAYER's `token.object.visible` was
  `false` (the player could not see it) while the GM still saw it. A robust, supported per-client
  hide. → Use the native `hidden` flag as the "invisible to players" base.
- **Show-to-spotted override:** on the player client, setting `token.object.visible = true` +
  `token.object.mesh.visible = true` DID render the hidden token — BUT the next visibility
  refresh (`refreshVisibility`) recomputed `visible = isVisible` and stomped it back to hidden.
- **Verdict:** the override must be REAPPLIED on every `refreshToken` / visibility refresh — the
  same hook-reapply pattern identity-mask uses for nameplates (and the planned 0.6.1 canvas art).
  Only public properties are touched (`visible` / `mesh.visible`), the same ones core assigns —
  no internal hacks.
- Caveats to handle during 0.2.0: the reapply timing (hook the visibility refresh), and verify no
  bad interaction with targeting / hover / fog for the shown token. The spike confirmed the core
  mechanism, not every edge case.

## Net effect on the plan

- **Decision #2 is confirmed viable:** native `hidden` base + reapply-on-refresh show-override is
  the chosen 0.2.0 approach.
- The two runtime-pending probes that gated Perception are resolved. The remaining probes
